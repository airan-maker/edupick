# Decision Log

> Updated: 2026-03-12

이 문서는 현재까지 대화에서 확정된 제품/구현 결정을 기록한다.
장기 전략 문서와 충돌할 경우, 현재 구현 우선순위와 실제 코드 작업 기준은 이 문서를 따른다.

## 1. 현재 제품 View 구조

### 결정
- 현재 서비스는 `학부모 View`, `강사 View` 두 흐름에 집중한다.
- `ADULT`, `ACADEMY_OWNER`는 현재 제품 범위에서 제외한다.

### 이유
- 실제 구현이 학부모 플로우와 운영 플로우 중심으로 진행되고 있었다.
- 성인 수강생과 학원장 전용 분리는 아직 제품 가치가 충분히 구현되지 않아 혼선을 만들었다.

### 적용 원칙
- 코드, 타입, DB 스키마, 온보딩, 설정 화면에서는 `PARENT`, `INSTRUCTOR`만 사용한다.
- 장기 전략 문서나 과거 아이디어 문서에는 `ADULT`, `ACADEMY_OWNER`가 남아 있을 수 있다.
- 장기 전략 문서의 존재가 현재 코드 구조를 다시 4역할로 되돌리는 근거가 되지는 않는다.

## 2. 역할 전환 정책

### 결정
- 개발 중에는 한 계정으로 `학부모 View`와 `강사 View`를 전환할 수 있게 둔다.
- 운영 배포 전에는 일반 사용자에게 역할 전환 UI를 계속 노출할지 다시 판단한다.

### 이유
- 구현 중에는 같은 계정으로 양쪽 화면을 빠르게 검증하는 편이 효율적이다.
- 현재는 QA와 기능 확인이 우선이다.

### 현재 상태
- 온보딩과 설정 화면에서 `학부모`, `강사 View` 두 옵션만 노출한다.
- 예전 `학원장`, `성인 수강생` 선택지는 제거했다.

## 3. 관리자/운영 화면 방향

### 결정
- 별도의 `학원장 View`를 두지 않는다.
- 운영 기능은 모두 `강사 View` 안에서 처리한다.

### 포함 범위
- 학원 관리
- 반 관리
- 공지 발송
- 수납 현황
- 운영 일정 확인

### 의미
- 기존에 학원장 전용으로 분리해 두었던 일부 관리 기능도 강사 View 안으로 흡수한다.
- 화면 명칭은 `운영 보드`, `스튜디오`, `강사 View` 등으로 정리한다.

## 4. 구현 우선순위

### 결정
- 현재 P0는 다음 두 축에 집중한다.
- `학부모 View`: 탐색, 신청, 캘린더, 결제, 마이페이지
- `강사 View`: 학원/반 운영, 공지, 수납

### 제외 또는 후순위
- 성인 수강생 전용 플로우
- 학원장 전용 별도 플로우
- 다중 역할 구조

## 5. 문서 반영 원칙

### 결정
- `ImplementationPlan.md`는 현재 실행 계획 기준으로 정리한다.
- 오래된 전략 문서나 장기 로드맵 문서는 보존할 수 있다.

### 현재 반영 상태
- `docs/ImplementationPlan.md`는 `학부모 View / 강사 View` 기준으로 정리했다.
- 장기 전략 문서에는 과거 확장 아이디어가 일부 남아 있을 수 있다.

## 6. DB / Enum 정책

### 결정
- `Role` enum은 `PARENT`, `INSTRUCTOR`만 남긴다.
- `ADULT`, `ACADEMY_OWNER`는 실제 코드와 DB에서 제거한다.

### 이유
- enum은 허용값이 고정된 타입이므로, 코드와 DB가 다른 상태로 남아 있으면 이후 개발과 데이터 검증이 꼬인다.

### 현재 상태
- 코드와 Prisma schema는 2역할 기준으로 정리했다.
- 기존 로컬 DB는 reset 후 다시 만드는 방식으로 맞춘다.
- 저장된 데이터는 중요하지 않으므로 DB 초기화가 허용된다.

### DB 재생성 원칙
- 기존 `edupick` DB는 삭제 후 재생성한다.
- 수정된 초기 migration으로 다시 올린다.
- 확인 쿼리 결과에서 `Role` enum 값은 `PARENT`, `INSTRUCTOR`만 보여야 한다.

## 7. 최근 구현 결정

### 운영 기능
- 공지 발송 기능을 placeholder가 아닌 실제 화면/API로 연결했다.
- 학원 관리 화면은 등록 + 수정 가능 상태로 정리했다.
- 반 관리 화면은 생성 + 수정 + 상태/시간표 수정 가능 상태로 정리했다.
- 수납 현황은 운영 요약 기반 화면으로 연결했다.

### 마이페이지/역할 UI
- 역할 저장만 되고 화면이 안 바뀌던 문제를 수정했다.
- 역할에 따라 하단 네비, 마이페이지 메뉴, 진입 경로가 달라지도록 정리했다.
- 현재 기준에서는 `학부모 View`와 `강사 View`만 남긴다.

### 캘린더
- 주간 일정과 월간 일정의 빈 상태 문구를 동일하게 맞췄다.

### 로그인 전 공개 구조
- 공개 메인 `/`를 one-page 서비스 소개 페이지로 정리했다.
- 학부모 실제 사용 홈은 `/home`, 강사 운영 진입은 `/instructor`, 로그인 후 강사 홈은 `/studio`로 유지한다.
- 공개 헤더 메뉴는 소개 구조 중심으로 두고, 앱 성격이 강한 메뉴는 기본 공개 메뉴에서 제외한다.
- 공개 페이지의 추천 학원/후기 영역은 실제 공개 API 응답을 사용하고, FAQ는 현재 구현 상태 기준 원문으로 관리한다.
